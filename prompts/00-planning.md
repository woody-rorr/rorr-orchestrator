# 플래닝 단계 (Planning Phase)

## 언제 실행하나

작업 의도가 있는 모든 요청에서 **도메인 MCP 호출 전에 반드시 플랜을 먼저 출력**한다.
단, 아래 경우는 플래닝 없이 즉시 처리:
- 단순 상태 조회 ("현재 배포 상태 알려줘" 등 read-only)
- 사용자가 이미 이전 플랜을 확인하고 "진행해줘" / "OK" / "맞아" 등으로 승인한 직후 turn

## 플랜 출력 형식

플랜은 아래 구조로 한국어로 작성한다. 항목이 해당 없으면 생략.

```
## 작업 플랜

### 요청 요약
(사용자가 원하는 것을 한 문장으로)

### 호출할 MCP
- backend-new: (이유)
- web: (이유)
- extension: (이유)
- notion: (이유)

### API 인터페이스 (백엔드 관련 시)
- `POST /api/...` — 요청: `{ ... }` / 응답: `{ ... }`
- `GET /api/...` — ...

### 데이터 모델 / 엔티티
- Entity: 필드 목록 및 타입

### 엣지 케이스 & 예외 처리
- (고려해야 할 케이스 나열)

### 모호한 부분 — 확인 필요
- ❓ (불명확한 요구사항 또는 결정이 필요한 항목)
```

## 플랜 출력 후 행동

1. "모호한 부분" 항목이 있으면 → 해당 항목에 대한 답변을 요청하고 **MCP 호출 중단**.
2. "모호한 부분" 항목이 없으면 → 플랜 출력 후 "위 플랜으로 진행할까요?" 한 줄 추가하고 **MCP 호출 중단**.
3. 사용자가 플랜을 승인하거나("진행해줘", "OK", "맞아" 등) 수정사항을 반영한 후에만 도메인 MCP를 호출한다.

## Notion 페이지가 있는 경우

Notion MCP를 먼저 호출해 페이지 내용을 가져온 뒤, 그 내용을 바탕으로 플랜을 작성한다.
Notion에서 불명확한 항목도 "모호한 부분"에 포함시킨다.

## 플랜 승인 후

확정된 플랜의 API 인터페이스·엔티티·엣지케이스를 각 도메인 MCP의 `user_message`에 상세히 포함시켜 전달한다.
MCP에 넘기는 메시지에 "플랜 확정 내용:" 접두어를 붙여 원본 요청보다 플랜을 우선하게 한다.
